# 收租吧交互重构方案（登录 + 四 Tab）

**版本：** v1.0  
**日期：** 2026-04-06  
**适用范围：** 小程序前端信息架构与导航交互（不改变领域数据模型）

## 1. 背景与目标

当前 `workbench` 页面同时承载首页盘面、提醒入口、业务维护、快速录入、退出登录等多个职责，存在以下问题：

- 页面信息密度过高，核心动作不聚焦
- 新用户首次进入路径不清晰，登录后“下一步做什么”不够明确
- 导航结构不稳定，维护功能与经营查看功能混杂

本方案目标：

1. 首屏明确为登录/注册入口（微信身份认证）
2. 登录后进入稳定的四栏主导航：`首页` / `业务维护` / `房源列表` / `我的`
3. 现有能力按职责归位，不新增平行状态语言
4. 保持与 Phase 3/4 的提醒与异常口径一致

## 2. 关键约束

- 状态语言保持统一：`即将到期`、`已逾期`、`空置过久`、`人工异常`（以及来自维修高频的异常事实）
- 维修异常仍接入现有异常链路，不单独新造一套状态体系
- 路由规范化：Tab 页之间只用 `switchTab`，非 Tab 功能页使用 `navigateTo`
- 不修改核心云端领域模型，仅做前端 IA 重组和页面职责拆分

## 3. 登录/注册定义

> v1 为单房东账号（微信身份），无需账号密码注册体系。

- “注册”定义：用户首次微信登录并建立本地会话
- “登录”定义：已有会话恢复，或再次微信授权登录

交互建议：

- 页面标题：`登录 / 注册`
- 主按钮：`微信一键登录`
- 辅助说明：首次使用将自动创建你的房东工作空间

## 4. 目标信息架构

### 4.1 顶层结构

- 未登录：`登录/注册页`
- 已登录：底部 TabBar（4 项）
  - `首页`
  - `业务维护`
  - `房源列表`
  - `我的`

### 4.2 Tab 页职责

1. 首页
- 展示经营盘面（概览卡、异常摘要、今日建议）
- 提供“异常处理”与“提醒中心”快捷入口
- 不放维护型长按钮列表（减少噪音）

2. 业务维护
- 聚合所有录入/建档动作：
  - 快速录入
  - 房源维护
  - 房间维护
  - 租户维护
  - 租约维护
- 作为“新增、编辑、维护”的唯一主入口

3. 房源列表
- 展示房屋/房间列表（搜索、筛选、状态与风险标签）
- 作为运营处理主战场，点击进入 `unit-detail`
- 从首页/提醒中心钻取时，沿用同一列表页 + 查询参数

4. 我的
- 账号与系统设置：
  - 提醒设置
  - 订阅状态说明
  - 退出登录
- 后续可扩展：导入模板下载、版本信息、问题反馈

## 5. 现有页面迁移映射

| 当前页面/功能 | 目标归属 | 处理方式 |
|---|---|---|
| `pages/workbench/index` 首页概览/异常/建议 | 首页 Tab | 保留并瘦身（移除维护入口区） |
| `workbench` 业务维护按钮组 | 业务维护 Tab | 拆到新 `ops` 页 |
| `workbench` 快速录入按钮 | 业务维护 Tab | 保留入口，落到 `quick-entry` |
| `pages/units/index` | 房源列表 Tab | 升为主 Tab 页 |
| `pages/reminder-settings/index` | 我的 Tab | 作为设置项二级页 |
| 退出登录 | 我的 Tab | 从 `workbench` 挪至“我的” |
| `pages/alerts/index` | 首页二级页 | 从首页/我的可进入 |
| `pages/unit-detail/index` | 房源列表二级页 | 保持详情闭环 |

## 6. 路由与页面建议

### 6.1 推荐页面布局

- `pages/auth/index`（登录/注册，非 Tab）
- `pages/home/index`（可由 `workbench` 重命名或直接替代）
- `pages/ops/index`（新建，业务维护）
- `pages/units/index`（Tab）
- `pages/profile/index`（新建，我的）

> 若希望最小改造，也可先保留 `workbench` 路径并作为“首页 Tab”，后续再重命名。

### 6.2 跳转规范

- 登录成功：`reLaunch` 到首页 Tab
- Tab 切换：`switchTab`
- 进入详情/表单：`navigateTo`
- 登出：清理会话后 `reLaunch` 到登录页

### 6.3 登录守卫

- `auth` 页：若已有会话，直接进入首页 Tab
- Tab 页 `onShow`：无会话则回登录页

## 7. 关键用户流程

### 7.1 首次使用

1. 打开小程序 -> 登录/注册页
2. 点击微信一键登录
3. 创建会话成功 -> 首页 Tab
4. 首页提示建议动作（如去业务维护完成建档）

### 7.2 日常经营

1. 首页查看异常与建议
2. 点击进入房源列表筛选集合
3. 进入详情完成账单/维修处理
4. 返回继续处理下一条

### 7.3 维护建档

1. 进入业务维护 Tab
2. 选择维护模块（房源/房间/租户/租约/快速录入）
3. 保存后返回业务维护或跳转相关列表

## 8. 分步改造计划（建议）

### Step 1：搭建骨架（低风险）

- 新增 `tabBar` 与 `ops/profile` 两个壳页
- `units` 接入 Tab
- `workbench` 先作为首页 Tab 使用

### Step 2：内容归位（中风险）

- 从 `workbench` 移除维护与退出入口
- 将维护入口搬迁到 `ops`
- 将提醒设置、退出登录搬迁到 `profile`

### Step 3：体验收敛（低风险）

- 统一按钮文案与状态词
- 清理冗余跳转路径
- 回归测试高频流程

## 9. 验收标准（UAT）

1. 未登录用户只能看到登录/注册页
2. 登录成功后进入四 Tab 结构，且 Tab 可稳定切换
3. 首页只保留经营盘面与处理入口，不再承担维护中心职责
4. 业务维护可访问所有建档动作
5. 房源列表可进入详情并完成账单/维修处理
6. 我的页面可管理提醒设置并执行退出登录
7. 首页/提醒中心/列表详情的异常口径完全一致
8. 不出现因路由重构导致的空白页、死链、循环跳转

## 10. 风险与规避

- 风险：Tab 页改造导致历史 `navigateTo` 到 Tab 报错  
  规避：统一替换为 `switchTab`

- 风险：登录态校验分散，产生闪屏或回跳  
  规避：统一封装会话守卫逻辑

- 风险：功能迁移后入口遗漏  
  规避：按“页面-能力对照表”逐项回归

## 12. 实施记录（2026-04-06）

### 12.1 已落地结果

- 登录/注册页保持 `pages/auth/index`，文案调整为“登录 / 注册”，并明确“微信一键登录”
- 登录后进入四 Tab 结构：`首页(workbench)` / `业务维护(ops)` / `房源列表(units)` / `我的(profile)`
- 首页已移除维护中心与退出登录，保留经营概览、异常摘要、今日建议、提醒入口
- 新增 `业务维护` Tab：快速录入、房源维护、房间维护、租户维护、租约维护
- 新增 `我的` Tab：提醒设置、订阅状态、账号信息、退出登录

### 12.2 路由收敛

- 登录成功与登出统一使用 `reLaunch`
- Tab 间切换统一使用 `switchTab`
- 二级页统一使用 `navigateTo`
- 首页/提醒中心跳转房源 Tab 时采用“临时存储筛选参数 + switchTab”方案，避免 Tab query 限制

### 12.3 约束对齐结果

- 异常状态口径保持统一：`即将到期` / `已逾期` / `空置过久` / `人工异常`
- 维修高频异常继续并入现有异常链路，不新增平行状态体系
- 前后端联调链路保留：首页聚合、提醒中心、房源列表、详情处理、维修记录均保持原能力
