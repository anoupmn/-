# 收租吧新手引导优化需求文档

版本：v1.0  
日期：2026-04-09  
范围：小程序前端交互与文案提示优化（不改领域模型、不改核心云端业务规则）

## 1. 背景

当前版本已经完成四 Tab IA 重构，结构上更清晰，但新手首次使用仍存在较高理解成本：

1. 首页偏经营盘面，不是首次建档入口，新手不知道“下一步先做什么”。
2. 建档链路被拆分在多个页面，步骤依赖关系需要用户自己推断。
3. 部分关键操作是图标或隐式入口，新手不容易识别。
4. 空状态文案未区分“还没数据”和“搜索无结果”两类场景。

## 2. 优化目标

1. 降低新手首次完成“房源 -> 房间 -> 租约”闭环的认知成本。
2. 把“功能导航”转化为“任务导航”，突出下一步动作。
3. 采用轻量提示和上下文引导，避免重型教程打断。
4. 保持老用户可快速操作，不牺牲现有高频路径效率。

## 3. 设计原则

1. 任务优先：先告诉用户下一步做什么，再解释功能概念。
2. 就地引导：提示出现在用户当前操作上下文内，减少跳读。
3. 小步可达：每一步结束后给明确“下一步”动作按钮。
4. 新老兼容：首次用户显示引导，已有数据用户继续看盘面。
5. 口径一致：状态词保持现有定义，不新增平行术语体系。

## 4. 目标用户与核心场景

### 4.1 目标用户

1. 首次使用的个人房东，尚未完成任何基础数据录入。
2. 低频使用、对当前 IA 不熟悉的回访用户。

### 4.2 核心场景

1. 登录后首次进入系统，不知道从哪个 Tab 开始。
2. 已新增房源，但不知道还需要添加房间才能录租约。
3. 已添加房间，但不知道应立即去租约录入。
4. 在房源列表看到空白时，无法区分“没数据”还是“筛选错了”。

## 5. 需求范围

### 5.1 本轮包含

1. 首页、业务维护、房源维护、房间维护、租约维护、房源列表、单元详情的提示文案与引导交互优化。
2. 首次建档任务引导卡（轻量，不做全屏强制教程）。
3. 关键动作的下一步推荐按钮或结果反馈增强。
4. 空状态分流文案与跳转动作优化。

### 5.2 本轮不包含

1. 云函数接口字段与领域模型调整。
2. 账单、提醒算法口径变更。
3. 权限体系重构与多角色模型。
4. 重型新手蒙层（全页面逐步遮罩教学）。

## 6. 页面级需求

### 6.1 登录页（auth）

1. 补充首次使用说明：首次登录会自动创建房东工作空间。
2. 登录成功后维持当前跳转逻辑，但在首页通过数据状态判定是否展示“首次引导卡”。

### 6.2 首页（workbench）

1. 当用户尚未完成基础建档时，顶部显示“首次建档引导卡”。
2. 引导卡包含 3 步任务：新增房源、添加房间、录入租约，并展示完成进度。
3. 引导卡提供主按钮跳转到当前应执行步骤。
4. 用户已有完整基础数据时，不显示引导卡，保持经营盘面优先。

### 6.3 业务维护（ops）

1. 增加“推荐建档顺序”区块，明确 1-2-3 步。
2. 每步提供动作按钮与一句目的说明。
3. 对前置未满足的步骤显示提示（例如未建房源时不能有效录入租约）。

### 6.4 房源维护（assets-form）

1. 保持现有表单结构，新增“保存成功后下一步”提示。
2. 将房源卡片上的加号行为显式化，补充“添加房间”可见文本。
3. 房源为空时提示“先新增房源”，并提供后续步骤预告。

### 6.5 房间维护（rooms-form）

1. 房间保存成功后，增加明显提示“下一步去录入租约”。
2. 提供直达租约录入页按钮，且带上当前房源上下文。
3. 当房源缺失时，提示返回房源维护，而不只提示失败信息。

### 6.6 租约维护（leases-form）

1. 维持当前“先选房源再选房间”逻辑，但强化前置条件提示。
2. 当无房源/无房间时显示对应引导文案和直达按钮。
3. 保存成功弹窗中增加“去房源列表查看详情”路径，形成闭环。

### 6.7 房源列表（units）

1. 区分两类空状态：
`首次空状态`：暂无任何可管理房源，引导去业务维护。
`搜索空状态`：当前筛选无结果，引导清空关键词。
2. 保持当前卡片信息密度，但入口提示文案更任务化。

### 6.8 单元详情（unit-detail）

1. 在账单、补录、维修关键动作旁增加轻量“何时用”说明。
2. 保持当前操作能力不变，降低术语理解成本。

## 7. 文案策略

1. 术语一致：沿用“房源、房间、租约、房源列表、单元详情”。
2. 行动导向：尽量使用“下一步：xxx”句式。
3. 低负担：一句话解决一个问题，不写长段说明。
4. 错误可恢复：失败提示尽量附带“应该去哪里修复”。

## 8. 数据与状态判断（前端）

首次建档判断需要最小数据判定能力：

1. 是否存在房源（assets > 0）。
2. 是否存在房间（rooms > 0，或可通过单元列表存在性替代判定）。
3. 是否存在有效租约（occupied 单元或 lease 数量 > 0）。

基于判定输出引导进度：

1. 0/3：优先引导新增房源。
2. 1/3：优先引导添加房间。
3. 2/3：优先引导录入租约。
4. 3/3：不再显示首次引导卡。

## 9. 验收标准（UAT）

1. 首次登录后，用户可在 30 秒内识别第一步入口。
2. 用户完成房源保存后，界面明确告知“下一步是添加房间”。
3. 用户完成房间保存后，界面明确告知“下一步是录入租约”。
4. 用户在房源列表空白时可明确理解是“没数据”还是“筛选无结果”。
5. 现有老用户进入首页时，不被强制教程打断。
6. 不引入新的路由死链、循环跳转或 Tab 跳转错误。

## 10. 风险与约束

1. 风险：引导信息过多导致页面噪音增加。  
约束：每页最多 1 个主提示区，次提示用短句。

2. 风险：前置判定依赖多数据源导致状态闪动。  
约束：先用已有 API 最小判定，必要时增加加载占位。

3. 风险：对老用户造成干扰。  
约束：首次引导卡只在未完成建档条件下展示。
