# 收租吧

## What This Is

收租吧是一个面向个人房东自用的微信小程序，用来管理大约 80 间房屋与房间的收租、租户、费用、维修/打理支出和异常状态。它的核心是让房东每天打开首页就能快速知道哪些房间快到期、哪些已经逾期、哪些空置过久、哪些存在维修或人工标记异常，并能在月底导出清晰的经营明细和收据。

## Core Value

让我每天打开小程序就能立刻知道该收谁的租、哪间房有异常、下一步该处理什么。

## Context

这是一个面向个人房东真实自用的小程序，业务对象是大约 80 间房，房型既有整套出租，也有一个房源下多个房间分别出租。首页最重要的是快速概览，尤其是 15 天内到期房屋数量、空置房屋数量和异常房屋列表。异常定义包括即将到期、逾期、空置过久、租客反馈维修、维修频次过高和人工标记。

数据层面需要从第一版就考虑历史可追踪性，包括历任租户信息、不同租约规则、分项账单、收款记录、维修/打理支出、退租支出、收据快照和月度导出。用户明确表示维修、灯具、墙面、厕所堵塞、保洁、请人打理等费用都是房东承担，录入目的是留痕、复盘问题房间和导出，不应进入租客应收账单。

原路线中的 Excel 导入与微信官方提醒消息实际发送闭环已删除。当前迭代方向调整为：先稳定费用与账单模型，再做月度导出与收据。

## Constraints

- **Platform**: 必须以微信小程序为主要交付形态 — 用户日常使用入口已经明确
- **Audience**: 第一版只服务单一房东账号 — 先把个人使用体验做扎实
- **Data Model**: 必须兼容“房源/资产 -> 房间 -> 租约”层级结构 — 同时适配整租和分租
- **Billing Integrity**: 已收账单、收据快照和历史维修/支出不得被静默删除或覆盖
- **Owner Expense Boundary**: 维修、保洁、打理等房东支出不进入租客应收账单，但需要支持统计和导出
- **UX**: 第一版采用黑字白底、功能优先的朴素风格 — 暂不把设计复杂度引入首轮开发
- **Implementation**: 优先采用成熟开源方案和官方能力 — 降低试错成本，避免重复造轮子

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| 第一版定位为个人房东自用工具 | 目标用户单一，优先把真实使用场景跑通 | 已落地（Phase 1-4） |
| 采用“房源/资产 -> 房间 -> 租约”作为核心模型 | 需要同时兼容整租和分房出租两种业务形态 | 已落地（Phase 1） |
| 费用模型从第一版开始支持分项账单 | 后续统计、提醒和对账都依赖细粒度费用结构 | 已落地（Phase 2），Phase 5 继续优化 |
| 微信提醒先只支持订阅引导和规则开关 | 实际发送闭环暂不做，避免偏离当前账本治理重点 | 已落地（Phase 3），发送闭环删除 |
| 首页优先展示 15 天内到期、空置数量与异常列表 | 这是用户每天打开后最关心的管理视图 | 已落地（Phase 3） |
| 异常建议先采用规则型触发 | 先保证建议稳定、可解释、可落地 | 已落地（Phase 3） |
| 维修记录采用固定分类 + 备注 | 兼顾统计分析和实际记录灵活性 | 已落地（Phase 4） |
| 维修/保洁/打理费用作为房东经营支出 | 用户只想留痕、复盘和导出，不应向租客应收 | 待落地（Phase 5） |
| 消防押金和锁卡押金作为一次性押金类应收 | 进入账单/收据/导出，但不参与逾期提醒或下一笔租金核心提示 | 待落地（Phase 5） |
| 月度导出采用一个 Excel 多 sheet | 同时满足日常算账、逐笔核对和支出复盘 | 待落地（Phase 6） |
