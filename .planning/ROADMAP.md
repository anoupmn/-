## 第 1 阶段：核心数据骨架

**目标：**
建立可长期演进的业务底座，保证系统从第一天起就能正确表达整租、分租、租户历史和租约生命周期，并完成不依赖导入的基础经营数据手动建档。

**Plans:** 4 plans

Plans:
- [x] 01-01-PLAN.md — 初始化微信小程序工程与测试基座
- [x] 01-02-PLAN.md — 实现微信登录、会话持久化与安全落点页
- [x] 01-03-PLAN.md — 固化核心领域契约并实现手动录入/查询后端通道
- [x] 01-04-PLAN.md — 交付快速录入、分步维护、经营列表与详情页闭环

**覆盖需求：**
`AUTH-01`, `AUTH-02`, `ASST-01`, `ASST-02`, `ASST-03`, `LEASE-01`, `LEASE-02`, `LEASE-03`, `LEASE-04`, `LEASE-05`, `LIST-01`, `IMPT-01`

**成功标准：**
1. 用户可以通过微信登录进入系统并保持登录状态
2. 用户可以创建房源、房间，并明确区分整租与分房出租
3. 用户可以创建租户和租约，并能结束租约而不丢失历史记录
4. 房屋/房间列表可以展示基础信息，支持日常浏览
5. 用户可以不依赖导入功能，手动录入房源、房间、租户与租约基础数据，分项费用仍留在第 2 阶段

**Post-debug polish captured after phase execution:**
- 房源维护页需要支持地区选择、地图选点失败兜底、详细地址补充与房源列表内直接添加房间
- 工作台恢复了业务维护入口，支持直接进入房源、房间、租户、租约与房态总览
- 房屋列表、房源列表、租约维护中的房源选择都需要模糊搜索和长列表折叠/展开
- 面向中国用户的页面文案必须中文化，避免直接暴露英文状态枚举或内部字段

## 第 2 阶段：账单与房态视图

**目标：**
让每个租约具备可计算、可追踪的分项账单和当前房态，支撑后续首页与提醒逻辑。

**Plans:** 3 plans

Plans:
- [x] 02-01-PLAN.md — 建立账单领域骨架、状态计算器与租约 bill 同步
- [x] 02-02-PLAN.md — 接通收款登记、详情账单分区与列表房态摘要
- [x] 02-03-PLAN.md — 升级小程序列表与详情经营视图

**覆盖需求：**
`ASST-04`, `BILL-01`, `BILL-02`, `BILL-03`, `BILL-04`, `LIST-02`

**成功标准：**
1. 每个租约都可以维护房租、押金、管理费等固定费用，水电/维修等浮动费用可在月度账单里逐月补录
2. 系统可以判断每一项费用的待收、已收和逾期状态
3. 房屋或房间详情页可以展示当前租约、应收信息和费用结构
4. 每个可出租单元都有稳定的当前状态表示

**Post-debug polish captured after phase execution:**
- 详情页账单视图从“费用分区”调整为“月度账单”，默认展开当前月，其他月份手风琴折叠
- 月度账单支持逐月补录水费、电费、维修费和其他浮动费用，补录项显示在每个月账单列表底部

- 房源维护支持直接删除测试房源，并级联清理房间、租约和账单，方便反复联调

## 第 5 阶段：费用与账单模型优化

**目标：**
让租约固定费用、水电抄表、房东维修/打理支出、账单纠错和集合写入边界更贴近日常收租使用方式，为后续月度导出与收据生成提供稳定数据基础。

**Plans:** 5 plans

Plans:
- [x] 05-01-PLAN.md — 数据集合职责、操作写入边界与账单同步安全治理
- [x] 05-02-PLAN.md — 固定费用、自定义费用与押金类一次性费用模型
- [x] 05-03-PLAN.md — 水电抄表计费、默认读数和月度账单展示
- [x] 05-04-PLAN.md — 维修/打理支出独立建模与房间问题分析口径
- [x] 05-05-PLAN.md — 退租、录错、删除/修改补救策略与全链路验证

**覆盖需求：**
`FEE-01`, `FEE-02`, `FEE-03`, `FEE-04`, `UTIL-01`, `UTIL-02`, `OPEX-01`, `OPEX-02`, `CORR-01`, `CORR-02`

**成功标准：**
1. 租约录入固定内置租金、押金、管理费、消防押金、锁卡押金五项
2. 消防押金和锁卡押金作为一次性押金类应收，进入账单、收据和导出，但不进入逾期提醒或下一笔租金核心提示
3. 管理费默认周期性，但允许在租约里改为一次性
4. 用户可以继续添加自定义费用，且必须选择费用性质：周期性费用、一次性费用或押金类费用
5. 水电费支持上期抄数、本期抄数和单价录入，并由服务端自动计算用量与金额
6. 维修、保洁、打理等费用作为房东经营支出独立记录，不进入租客应收账单
7. 维修类支出参与房间问题分析；保洁、打理、请人管理等支出不计入维修高频异常
8. 编辑租约时只替换未收款的系统生成账单，已收账单和手工账单不得被静默删除
9. 租约支持安全删除；一旦产生已收账单、收据或维修/支出关联，则禁止硬删除，只能编辑、更正、结束或作废

**UI hint**: yes

## 阶段排序依据

1. 房源/房间/租约模型必须先稳定，否则后面的账单、提醒和维修统计都会反复返工
2. 分项账单和房态必须先完成，首页和提醒才有可靠的数据来源
3. 首页概览与提醒是 v1 的核心价值，所以在主数据稳定后优先交付
4. 维修统计依赖历史租约和稳定详情页，因此放在提醒闭环之后更稳
5. 费用与账单模型必须先完成调整，导出和收据才能引用稳定口径
6. 月度导出与收据依赖新的费用分类、水电计算、房东支出和退租补救口径

## 不再纳入当前路线

- Excel 导入房屋、租约、租户及账单数据：本轮明确删除，不作为当前版本后续阶段
- 微信官方提醒消息实际发送闭环：本轮明确删除，不作为当前版本后续阶段
- 家人或合伙人提醒接收：仍属未来多人协作能力，当前个人自用版本不做
