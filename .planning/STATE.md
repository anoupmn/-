## 本次决策

- 核心业务模型固定为 `asset -> room -> lease`，整租也通过默认整租单元统一表达
- 页面层统一通过 service 调用云函数，不直接操作集合或推导房态
- Phase 2 将在当前详情与列表结构上补账单和房态细节，而不是重做数据骨架
- 在真实联调中确认：房源、房间、租约录入必须围绕“先选房源再补下游信息”的自然链路组织，不能暴露内部 ID 给用户
- 长列表默认折叠并辅以模糊搜索，比单纯下拉滚动更适合 80+ 房源场景
- 地图选点在模拟器里不稳定，地址录入需要保留地区与详细地址的手动兜底路径
- Phase 2 以 `bills` 集合承接账单真相，列表与详情统一消费共享状态聚合结果
- 收款动作只更新 bill，不回写租约字段，历史账单状态由 `receivedAt` / `receivedAmount` 驱动
- 真实联调后确认：租约里只维护固定费用，水电/维修等浮动费用必须落在“单元详情 -> 月度账单”里逐月补录
- 单元详情的账单阅读顺序以“系统自动生成固定项在前，手动补录项在后”为准，减少收租时的视觉混乱
- 房源维护已补上删除能力，便于清理联调产生的测试房源、房间、租约与账单脏数据
- 首页概览、提醒中心和主建议统一复用共享 `alert evaluator` 结果，避免口径漂移
- 人工异常通过 `abnormal_flags` 独立持久化，不回写房间真相字段
- 首页异常数按受影响房间去重统计，提醒中心继续按规则类型分组
- 首页与提醒中心统一钻取到现有 `units` / `unit-detail`，不新增平行筛选页
- 页面层采用预生成 URL 的导航模式，避免在小程序 dataset 里传对象造成跳转不稳定
- 提醒偏好采用 `consentState + hasRequested + enabledRuleTypes` 持久化，首页只做一次性引导
- 提醒设置页只提供规则类型开关，不扩展阈值编辑、收件人管理或发送日志
- 提醒模板 ID 改为 `miniprogram/config/notification.*` 配置输入，避免在 service 文件硬编码占位值
- 工作台已补齐业务维护入口，支持直接进入房源、房间、租户、租约和房态总览
- 维修记录采用固定分类 + 备注，发生日期作为租住期间自动关联与统计的唯一时间锚点
- 维修高频异常沿用 Phase 3 既有异常口径（`abnormal_flags` + 统一提醒聚合），不新增平行状态体系
- 详情页历史区固定支持“历史租户 + 历史维修 + 每任租户期间维修次数”联合追溯
- 交互重构采用“最小迁移”策略：保留 `workbench` 路径作为首页 Tab，新建 `ops` 与 `profile` Tab，减少联调破坏面
- 首页/提醒中心跳转房源 Tab 的筛选参数改为“临时存储 + switchTab 消费”，规避 Tab 不支持 query 的限制
- 房源列表 Tab 增加 `onTabItemTap` 过滤复位，保证手动点击 Tab 时可快速回到全量视图
- 当前路线删除 Excel 批量导入和微信提醒实际发送闭环，优先继续优化手动录入后的经营闭环
- 第 5 阶段必须先审计数据集合职责和各操作对应的数据修改，发现读写冲突要优先修正
- 固定费用新增消防押金和锁卡押金，均为一次性押金类应收
- 自定义费用必须选择性质：周期性费用、一次性费用或押金类费用
- 水电费改为抄表计费：上期读数、本期读数、单价，金额由服务端计算
- 维修、保洁、打理、请人管理等费用均属于房东经营支出，不进入租客应收账单
- 房东支出金额可选；填了金额后进入支出统计和月度导出
- 只有维修类支出参与房间问题/高频维修异常分析，保洁、打理、请人管理不计入问题频次
- 租约录错支持安全删除；一旦产生已收账单、收据或维修/支出关联，则禁止硬删除
- 月度导出采用一个 Excel 多 sheet，支持全部房源导出和单房源/房间筛选导出
- 月度明细参考“每月房租、水电收入及各项支出”表头，一行一个房间
- 收据只针对租客已收款项，不包含房东支出；收据快照固定，误开时删除收据并解除账单引用后重新开具
- 收据闭环最终口径：按租约月份生成一张收据，不按账单细项逐张生成；收据记录按月份自动整理，可筛选、查看、回房间、删除和导出 PDF 打印版
