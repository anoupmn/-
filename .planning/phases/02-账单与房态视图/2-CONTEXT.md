# 第 2 阶段：账单与房态视图 - 上下文

**收集时间：** 2026-04-03
**状态：** 可进入规划

<domain>
## 阶段边界

本阶段只交付“账单与房态视图”，也就是在既有 `房源 -> 房间 -> 租约` 骨架上，补齐分项账单、费用状态、房态口径和详情页当前经营信息展示。

本阶段不包含首页概览、应用内提醒、微信提醒发送闭环、维修记录与异常分析、Excel 导入，也不提前引入人工标记异常。这些能力属于后续阶段。

本阶段要解决的是：让每个租约具备可计算、可追踪的费用记录，让每个可出租单元具备稳定一致的当前状态表达，并让详情页优先服务“今天该收什么、哪里异常、下一步做什么”的日常管理场景。

</domain>

<decisions>
## 实现决策

### 账单结构
- **D-01：** 费用结构采用“固定大类 + 可追加其他费用”的模式。
- **D-02：** 默认固定费用大类至少包括：房租、押金、水费、电费、物业费。
- **D-03：** 除固定大类外，允许在租约下追加多个“其他费用项”，用于覆盖少量特殊费用。
- **D-04：** 系统不仅维护租约上的费用规则，还要生成每期具体应收记录，后续状态、详情和提醒均以具体记录为准，而不是只在前端临时推导。

### 费用状态规则
- **D-05：** “已收”必须通过录入收款日期和收款金额产生，不使用纯布尔标记。
- **D-06：** Phase 2 不引入“部分收款”状态；一笔费用在录入收款日期和金额后即视为已收。
- **D-07：** 押金不纳入逾期心智；房租、水电、物业、杂费和追加的其他应交费用可以进入逾期状态。
- **D-08：** 费用状态粒度固定为：`待收 / 今日到期 / 已收 / 逾期`。

### 房态口径
- **D-09：** 房态采用“主状态 + 风险标签”的表达方式，而不是把所有状态压成单一互斥枚举。
- **D-10：** 主状态只表达占用关系：`已出租 / 待入住 / 空置`。
- **D-11：** 风险标签单独表达经营风险：`即将到期 / 已逾期 / 异常`。
- **D-12：** “即将到期”的统一阈值为 15 天内，和首页未来要使用的时间窗口保持一致。
- **D-13：** Phase 2 内“异常”先只由逾期触发；人工标记异常不提前纳入本阶段范围。
- **D-14：** 单元可以同时呈现主状态和风险标签，例如“已出租 + 即将到期”或“已出租 + 已逾期 + 异常”。

### 详情页信息层级
- **D-15：** 详情页采用“混合摘要优先”的第一屏结构，先展示当前经营摘要，再进入租约和账单细节。
- **D-16：** 第一屏摘要必须优先展示：主状态、风险标签、当前租户、下一笔应收日期、下一笔应收金额、逾期提示。
- **D-17：** 第一屏高频动作固定为：`登记收款` 和 `查看全部账单`。
- **D-18：** 历史租约和历任租户仍保留在详情页，但默认折叠，当前经营信息优先。

### 非房租费用展示
- **D-19：** 费用展示按三块分区：`房租`、`押金`、`非房租类费用`。
- **D-20：** “非房租类费用”统一承载水费、电费、物业费、杂费以及追加的其他费用项。
- **D-21：** 不为追加费用开放用户自定义归类；追加项默认归入“非房租类费用”。

### 与前序阶段保持一致
- **D-22：** 继续沿用 `房源 -> 房间 -> 租约` 作为唯一核心事实来源，不新增与租约平行的“当前账单摘要真相字段”。
- **D-23：** 页面层继续保持 `page -> service -> cloud function` 的薄页模式，不在页面端自行推导房态或账单真相。
- **D-24：** 列表与详情继续优先服务“每天扫一眼就知道该收谁、哪间房异常”的房东经营场景。

### the agent's Discretion
- 分项费用录入表单的具体排版和分组方式
- 详情页摘要卡的具体视觉层次与字段标签文案
- “今日到期”与“逾期”在列表/详情中的颜色或标签样式
- 账单列表按时间、类型或状态的具体排序细节

</decisions>

<specifics>
## 具体想法

- 账单结构不能只停留在租约规则层，必须有每期具体应收记录，这样后续提醒、首页和导入才能统一使用同一套事实。
- 房态不应该把“已出租”和“即将到期/逾期”互相覆盖；占用事实和经营风险必须并行表达。
- 详情页点进去的第一目标不是回顾历史，而是快速确认“当前是谁住、下一笔该收什么、有没有逾期”。
- 押金在经营心智上不同于周期性收租，因此单独分区展示比硬并入“房租类”更清晰。

</specifics>

## 现有代码洞察

### 可复用资产
- `cloudfunctions/shared/calculators/lease-lifecycle.ts`：已提供租约生命周期和“下一次应收日”推导雏形，可作为账单生成和房态判断的基础。
- `cloudfunctions/shared/calculators/rentable-unit.ts`：已存在单元摘要计算器，当前能产出 `currentStatus`、`nextReceivableDate`、`nextReceivableAmount`、`overdueDays`、`vacancyDays`，适合作为 Phase 2 房态扩展落点。
- `cloudfunctions/rentable-unit-detail/index.ts`：详情云函数已能返回 `activeLease`、`leaseHistory`、`tenantHistory`，后续可在此基础上补充当前账单与费用分区信息。
- `miniprogram/pages/units/index.ts` 与 `miniprogram/pages/unit-detail/index.ts`：列表页和详情页的页面壳子已存在，可在不重做导航结构的前提下扩充账单与房态展示。

### 已建立模式
- 页面继续通过 service 调云函数，不直接读取集合或自行拼装经营真相。
- 列表和详情优先消费后端派生字段，而不是把复杂规则留给前端。
- 当前代码中的房态雏形是单枚举 `occupied / pending_move_in / vacant / overdue`，Phase 2 需要在这个基础上升级为“主状态 + 风险标签”。

### 集成点
- 账单记录会成为后续首页概览、到期/逾期提醒、月度导出和收据的共同数据来源。
- 详情页当前租约区域需要与新增账单结构直接集成，而不是新开平行详情页。
- `lease` 当前仅有 `rentAmount` 与 `depositAmount` 字段，Phase 2 需要扩展出适配固定费用大类和追加费用项的结构，同时保持对既有租约流程的兼容。

</code_context>
