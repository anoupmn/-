# Phase 03 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none |
| Icon library | none |
| Font | WeChat Mini Program default sans stack |

### Phase-specific UI contract

- Keep the existing product language: plain white surfaces, dark text, light gray secondary surfaces, blue used only for operational emphasis.
- The home page replaces the current `workbench` placeholder and becomes a landlord operations dashboard, not a marketing-style landing page.
- The visual goal is operational clarity, not ornament. Strong hierarchy should come from spacing, section order, chip states, and amount typography rather than decorative chrome.
- The dashboard must feel consistent with existing `units` and `unit-detail` pages so the user perceives one continuous workflow.

---

## Layout Contract

### Page structure

The home dashboard uses a single-column vertical flow:

1. Page heading and current-period status copy
2. Three overview cards in reading priority order
3. Abnormal list section
4. Single primary recommendation card
5. Reminder subscription prompt or reminder settings entry

### Mobile behavior

- Optimize for one-hand scanning on WeChat mobile screens.
- Keep all critical summary content visible without horizontal scrolling.
- The first screen should show at least the heading plus the three overview cards on common devices.
- Do not use dense data tables or multi-column management layouts in this phase.

### Drill-down behavior

- Every overview card must be tappable and navigate directly to a filtered actionable list.
- Every abnormal list row must expose a direct path to either filtered unit list or unit detail.
- The recommendation card must open the exact filtered list that matches the recommended action.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Micro gaps inside pills and inline metadata |
| sm | 8px | Compact vertical rhythm inside cards |
| md | 16px | Default content gap between label/value pairs |
| lg | 24px | Card padding and section spacing |
| xl | 32px | Major gap between dashboard sections |
| 2xl | 48px | Top page breathing room and section resets |
| 3xl | 64px | Reserved for larger phase transitions or future full-screen empty states |

Exceptions: pill radius may stay fully rounded; Mini Program `rpx` equivalents may be used as long as they preserve the same scale ratio already present in `units` and `unit-detail`.

### Spacing rules

- Reuse the existing `24rpx` card padding and section gap as the default dashboard rhythm.
- Avoid mixing more than three vertical gap sizes in one section.
- The abnormal list and recommendation card should visually sit closer to each other than to the overview-card row, reinforcing the “see issue -> act” flow.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.6 |
| Label | 12px | 700 | 1.4 |
| Heading | 17px | 700 | 1.4 |
| Display | 22px | 700 | 1.2 |

### Typography rules

- Use `Heading` for section titles such as “首页概览”, “异常房间”, “今日建议”.
- Use `Display` only for the three overview numbers and key monetary/priority figures.
- Use `Label` for support text like “15 天内到期” and reminder-type headers.
- Never use display-sized typography for decorative emphasis or secondary metadata.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #ffffff | Page background and primary card surfaces |
| Secondary (30%) | #f9fafb | Secondary cards, grouped sections, soft data blocks |
| Accent (10%) | #2563eb | Tappable primary actions, status emphasis, selected summary card state |
| Destructive | #dc2626 | Overdue severity text, destructive actions, high-risk abnormal emphasis |

Accent reserved for: primary CTA buttons, active navigation target, selected reminder group, important count emphasis, and tappable recommendation entry. Never use accent blue for all pills, all labels, or generic body text.

### Semantic color contract

- Normal supporting text: `#4b5563`
- Strong primary text: `#111827`
- Neutral status chip background: `#f3f4f6`
- Informational status chip background: `#dbeafe`
- Danger border/background accents: `#fecaca` / white surface with red text

### Risk display rules

- “已逾期” and destructive actions use the destructive palette.
- “即将到期” may use accent blue emphasis, but must remain visually weaker than overdue.
- “空置过久” should use neutral emphasis first; do not style it as destructive unless product logic explicitly upgrades it to abnormal.
- Artificially colorful dashboards are out of contract for this phase.

## Component Contract

### Overview cards

- Exactly three cards at the top level: `15 天内到期`, `当前空置`, `异常`.
- Each card contains:
  - count as the dominant value
  - one short label
  - optional secondary hint such as “较昨日 +1” only if data exists; otherwise omit
- Cards are tappable blocks, not static metrics.
- The abnormal card may carry slightly stronger emphasis than the other two, but should not visually overpower the page.

### Abnormal list

- The abnormal list is the first detail section below the overview cards.
- Each row should show:
  - unit display name
  - primary abnormal reason
  - one short support line, such as tenant name, overdue days, or vacancy days
- Limit default first-screen rows to a compact set; overflow should use “查看全部” rather than an infinitely long home list.
- If a unit matches multiple abnormalities, show one primary reason in the row and defer full detail to drill-down.

### Recommendation card

- Show one primary recommendation only.
- Structure:
  - small label: `今日建议`
  - one action-oriented sentence
  - optional reason line with affected count
  - one clear CTA affordance
- The card should feel actionable, not analytical. Avoid paragraph copy.

### Reminder grouping view

- Reminder center groups by rule type, not by unit.
- Group headers should mirror the context decisions: `即将到期`, `已逾期`, `空置过久`, `人工异常`.
- Use expandable grouped sections or stacked cards, but keep the first level focused on rule category counts.

### Reminder settings entry

- On the home page, show reminder subscription state as a lightweight prompt or settings entry, not as a full configuration matrix.
- Full rule toggles belong in a dedicated settings page reachable from home.
- The settings page should expose rule-type switches only; do not introduce threshold editing or recipient management in this phase.

---

## Interaction Contract

### Primary flow

- The expected daily flow is:
  1. scan the three overview cards
  2. inspect the abnormal list
  3. take the suggested action
- The UI should never force the user to enter settings or reminder history before they can see today’s operational state.

### Empty states

- If there are no current issues, replace the abnormal list with a calm positive state rather than a blank hole.
- Empty-state copy should confirm the current healthy state and point to a sensible next action such as checking all units or managing reminder rules.
- If reminders are not yet subscribed, the home page may still show operational issues; subscription setup must not block dashboard usage.

### State transitions

- Tapping a summary card always keeps the user in the operational flow by taking them to a filtered list.
- Tapping a row in the abnormal list goes deeper to the relevant list or unit detail.
- Tapping the recommendation card should never open a passive explanation page.

### Loading and density

- Prefer skeleton or reserved card blocks over spinner-only loading.
- Avoid stacking more than two lines of metadata inside one abnormal row.
- If data volume is high, truncate on home and route to dedicated lists rather than making home scroll-heavy.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | 立即处理 |
| Empty state heading | 当前暂无异常 |
| Empty state body | 今日没有到期、逾期或异常房间。你可以继续查看全部房屋或检查提醒设置。 |
| Error state | 加载首页失败，请下拉重试或稍后再试。 |
| Destructive confirmation | 关闭提醒: 关闭后你将不再收到该类提醒，可随时在设置中重新开启。 |

### Copy rules

- All user-facing copy stays in Chinese.
- Recommendation copy must start with an action verb, for example `优先处理`、`先联系`、`先查看`.
- Avoid abstract wording like “系统检测到若干异常”; prefer concrete wording tied to the next action.
- Overview labels and reminder groups must use the same wording across home, reminder center, and settings.
