# 第 3 阶段：首页概览与提醒引擎 - 上下文

**收集时间：** 2026-04-03
**状态：** 可进入规划

<domain>
## 阶段边界

本阶段只交付“首页概览与提醒引擎”，也就是把首页工作台升级为真正的经营驾驶舱，围绕既有的账单、房态和单元摘要结果，补齐首页概览、应用内提醒、规则型建议，以及提醒订阅与规则开关设置。

本阶段不包含微信提醒消息的实际发送闭环、维修频次异常分析、Excel 导入，也不扩展到家人或合伙人接收提醒。维修频次异常属于第 4 阶段；微信实际发送闭环与 Excel 导入已在 2026-04-27 路线更新后从当前路线删除。

本阶段要解决的是：让房东每天打开首页，先看到全局概览，再看到最值得立刻处理的问题，并且能够一键进入对应列表继续处理。

</domain>

<decisions>
## 实现决策

### 首页第一屏结构
- **D-01：** 首页第一屏采用“数字概览卡 + 异常列表 + 今日建议”的驾驶舱结构，而不是待办清单优先。
- **D-02：** 顶部概览固定展示三类数字：`15 天内到期数量`、`当前空置数量`、`异常数量`。
- **D-03：** 第一屏的信息阅读顺序固定为：先看全局数字，再看异常列表，最后看建议动作。

### 提醒中心分组方式
- **D-04：** 应用内提醒默认按规则类型分组，而不是按房屋或房间分组。
- **D-05：** Phase 3 的提醒类型至少包括：`即将到期`、`已逾期`、`空置过久`、`人工异常`。
- **D-06：** 首页概览、异常列表和提醒中心要尽量共用同一套风险口径，降低每天切换视角的认知成本。

### 规则型建议表达
- **D-07：** 首页默认只突出一条“主建议”，不并列展示多条建议卡。
- **D-08：** 建议必须是可执行动作，而不是纯描述性文案；点击建议后直接进入对应筛选列表。
- **D-09：** 建议的目标是告诉用户“现在先做什么”，而不是重复展示原始状态统计。

### 订阅提醒入口与粒度
- **D-10：** 订阅提醒采用“首页一次引导 + 设置页长期管理”的模式。
- **D-11：** 订阅后的提醒管理按规则类型开关，不在本阶段细化到更复杂的阈值或收件人层级配置。
- **D-12：** 首页可以承担首次订阅引导，但持续性的规则管理入口应落在独立设置页，而不是把复杂配置长期放在首页。

### 与前序阶段保持一致
- **D-13：** 继续沿用 `asset -> room -> lease -> bills` 作为首页、提醒和建议的唯一事实来源，不新增平行真相字段。
- **D-14：** 页面层继续保持 `page -> service -> cloud function` 的薄页模式，首页和提醒页消费后端聚合结果，不在页面端自行拼装经营判断。
- **D-15：** “15 天内到期”继续沿用第 2 阶段已经锁定的统一窗口，首页、提醒和建议保持同口径。
- **D-16：** 首页交互目标仍然是“每天扫一眼就知道该收谁、哪里异常、下一步做什么”，不向报表或长篇分析页演化。

### the agent's Discretion
- 异常列表默认展示的条数、排序细节和字段排布
- 主建议的优先级打分细则与同分时的兜底规则
- 首页概览卡、提醒分组列表和设置页的具体视觉层次与按钮文案
- 提醒历史是否保留部分已处理记录，以及保留时长和折叠方式

</decisions>

<specifics>
## 具体想法

- 首页要先给房东“全局盘面感”，所以第一眼应该先看到到期、空置和异常这 3 个数字，而不是先埋进待办细节。
- 提醒中心更适合按问题类型组织，这样每天处理节奏更像“今天先清逾期，再看空置，再看人工异常”。
- 规则建议不应该像运营报告，而应该像一句可以立即执行的行动提示，例如“优先处理 3 笔已逾期账单”。
- 订阅提醒第一版只做轻量开关和订阅入口，不提前演化成复杂的通知配置系统。

</specifics>

## 现有代码洞察

### 可复用资产
- `cloudfunctions/shared/calculators/rentable-unit.ts`：已经能产出 `mainStatus`、`riskTags`、`summaryHint`、`overdueDays`、`vacancyDays`，适合作为首页概览与异常列表的基础摘要来源。
- `cloudfunctions/shared/constants/statuses.ts`：已固化主状态和风险标签常量，可直接复用到提醒类型和首页标签文案。
- `cloudfunctions/rentable-units-list/index.ts` 与 `cloudfunctions/rentable-unit-detail/index.ts`：已具备列表与详情级摘要聚合逻辑，可作为 Phase 3 首页聚合和 drill-down 的承接点。
- `miniprogram/pages/workbench/index.*`：当前仍是已登录后的占位工作台，是最自然的首页驾驶舱落点。
- `miniprogram/pages/units/index.*` 与 `miniprogram/pages/unit-detail/index.*`：现有房屋列表和详情页已经具备“状态 + 风险 + 下一笔应收”的经营阅读模式，可作为首页跳转目的地和交互参照。

### 已建立模式
- 页面统一通过 service 调用云函数，不直接读取集合或在前端推导经营真相。
- 列表与详情已经采用“主状态 + 风险标签 + 提示文案”的阅读模式，首页应延续这套心智而不是重新造一套状态语言。
- 当前代码里尚未出现 alerts、订阅提醒或提醒设置的实现，本阶段会是这部分能力的第一次正式落地。

### 集成点
- 首页概览、异常列表和主建议需要消费同一套后端聚合结果，避免同一房间在不同模块里口径不一致。
- 提醒中心需要和现有房屋列表或详情页打通，支持从提醒项快速进入对应筛选列表或房间详情。
- 订阅设置需要承接微信订阅授权入口；真正的消息发送闭环已在 2026-04-27 路线更新后从当前路线删除，因此本阶段只保留“订阅能力与规则配置”的历史口径。
