# 第 4 阶段：维修档案与异常分析 - 上下文

**收集时间：** 2026-04-06
**状态：** 可进入规划

<domain>
## 阶段边界

本阶段只交付“维修档案与异常分析”，即：补齐维修记录建档、租住期间关联、维修统计与高频异常识别，并把这些结果接入现有详情页、首页与提醒中心。

本阶段不扩展微信模板消息发送闭环（Phase 5）、Excel 导入（Phase 5）、多人接收提醒（v2）。

本阶段的异常能力必须复用 Phase 3 既有异常链路与状态语言（同一套 `abnormal` bucket、同一提醒链路），不新增平行异常体系。

</domain>

<decisions>
## 实现决策

### 维修记录建模
- **D-01：** 新增独立维修记录事实集合，支持记录到 `room`（房间）或 `asset`（整套/资产）。
- **D-02：** 维修分类采用固定枚举：`水路`、`电路`、`家电`、`结构`、`安全`、`其他`，不开放自定义分类。
- **D-03：** 每条维修记录必须包含分类与备注，备注作为后续追溯与异常解释的主文本来源。
- **D-04：** 维修记录必须包含发生日期（`occurredAt`），统计与异常识别统一按发生日期计算。

### 与租住期间关联
- **D-05：** 创建维修记录时，系统按 `occurredAt` 自动匹配对应租住期间并回填 `leaseId`、`tenantId`（能匹配则写入，不能匹配则为空）。
- **D-06：** 租住期间判定基于租约时间窗（`startDate <= occurredAt <= endDate`），且不依赖前端传入租约 ID。
- **D-07：** 详情页展示维修历史时，优先展示关联租户名与租约起止，缺关联时明确展示“未关联租住期间”。

### 统计口径
- **D-08：** 每房总维修次数统计维度固定为“当前单元（room）全历史维修条数”。
- **D-09：** 每任租户期间维修次数由“该租约时间窗内的维修记录数”计算，不做前端推导。
- **D-10：** 主要维修类别按房间历史维修记录分组计数，按次数倒序返回前 N（默认 3）。

### 高频异常识别与接入
- **D-11：** 高频维修异常阈值固定为“近 30 天维修次数 >= 3”。
- **D-12：** 高频异常结果接入现有异常链路（`abnormal_flags` + alert evaluator），不新建平行 alert 类型体系。
- **D-13：** 提醒中心与首页继续使用 Phase 3 的规则型分组与异常 bucket；维修异常通过既有异常类型通道展示。
- **D-14：** 异常说明文案必须显式带出维修统计依据（如“近 30 天维修 3 次”），避免黑盒判断。

### 详情页呈现（LIST-03）
- **D-15：** 详情页历史区扩展为“历史租户 + 历史维修”并行展示，默认折叠，展开后可读。
- **D-16：** 详情页新增维修统计摘要（总次数、近 30 天次数、主要类别）和维修记录录入入口。
- **D-17：** 历史租约展示必须补齐租户名，不再只显示 `tenantId`。

### 可部署与稳定性约束
- **D-18：** 新增/改造云函数继续采用“函数目录内自带 `shared`”的可独立部署结构。
- **D-19：** 维修相关集合缺失时自动创建，避免首次联调失败。
- **D-20：** 任何集合清理逻辑不得使用空查询删除；更新逻辑不得写入 `_id`。

### the agent's Discretion
- 维修记录在详情页中的排序细节（默认按发生日期倒序）
- 维修类别统计可视化样式（文本列表或标签）
- 高频异常摘要在首页/提醒中心的文案精简策略

</decisions>

<specifics>
## 具体想法

- 本阶段必须让“维修异常”成为与收租异常并列的经营信号，但口径和状态语言要延续 Phase 3。
- 维修统计要服务日常决策：先知道是否高频异常，再知道集中在哪类问题、发生在谁租住期间。
- 详情页应把“历史租户 + 历史维修”放到同一阅读路径，减少跨页追溯成本。

</specifics>

## 现有代码洞察

### 可复用资产
- `alert-evaluator` + `dashboard` 已完成异常聚合与优先级排序，可作为维修异常接入口。
- `abnormal_flags` 已作为异常事实集合落地，可复用为维修高频异常持久化通道。
- `rentable-unit-detail` 已提供 `leaseHistory`、`tenantHistory`、月度账单结构，适合扩展维修档案与统计。

### 已建立模式
- 页面层持续采用 `page -> service -> cloud function`，前端不直接读集合。
- 首页与提醒中心以统一 alert 事实为来源，异常统计按房间去重。
- 云端已建立“缺集合自动创建、避免空查询删除、避免 `_id` 更新”的安全模式。

### 集成点
- 维修记录需要进入详情聚合（`rentable-unit-detail`）并参与异常评估（`dashboard-home` / `alerts-list`）。
- 提醒设置继续沿用既有规则类型开关，不新增平行设置页或第二套异常开关。
- 单元列表 `bucket=abnormal` 的钻取逻辑必须继续生效，维修异常也需可钻取。

</code_context>
